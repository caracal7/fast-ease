Quick Start

💡 Core idea

Time is money. Time to market is the way to money. We don’t have the money to lose our time with slow dev tools.☝️ 🙂

Key benefits

  • Develop faster. No dev server, no bundler, no toolchain, no bootstrap template. Starts up in 0 ms 🔥
  • No need for any other dependencies for building UI
  • Use blazing fast virtual DOM over native Web Components
  • True reactive 2-ways state exchange between components
  • Declare new components right inside HTML. Try any ideas right now! 🤘
  • Use nice template DSL as similar to HTML and JavaScript as possible
  • Unleash the power of WASM in JavaScript manner

i.js aims to provide a tool for development and application architecture out of the box. It empowers an effortless creation of highly scalable and easily maintainable applications without Babel transpiler, Parcel, Webpack and any complicated toolchain. All you need is i.js and a modern browser with customElements support (AD 2020+).

Before you start

You should be familiar with major modern yet fundamental web technologies 🤗
  • HTML5 - visual structure
  • CSS3 - styling
  • JavaScript ES2018+ - application logic

If you are not... Well, it is still possible but your way would not be an easy one and your components would be ugly 💩

👋 Hello i.js

What happened?

Looks short and simple but how does it work under the hood? 🧐

It may not look so easy at the beginning but by the end of Quick start section you will get a clear understanding of i.js basics. Hopefully.

Let’s get into some details.

Using <i-js> tag with no attributes means that we declare a component and immediately insert it into DOM with a pseudo-random name based on the component source hash. It looks something like <iris-1028423691-1191028423691>. Such inline component can only be used as a top-level application component, for prototyping or examples like this one.

Template is the most important part of any visual component. Template must be a top-level child of <i-js> tag and is declared inside <script type="iris/template"> tag. Template itself is some kind of DSL and a superset of HTML. It’s very similar to HTML yet has some differences. For example <div class="myclass"> is HTML but <div class=myclass> is not. We will talk about it later.

Our micro template is

<input %value=state.name> Hello ${state.name}

Template syntax is not of any importance now, do no try to remember it.

Reusing components

Let’s add a name to our component. In this case the name would be my-first-component.

The first attribute name of <i-js> tag becomes a new component’s name. Value of this atrribute is ignored. <i-js my-first-component=”foobar”> and <i-js my-first-component> both result in declaration of a native WebComponent called my-first-component. It is not immediately inserted into DOM like an inline component right after declaration and can be used anywhere in a page like a regular HTML tag. Because i.js is based on native Web components the naming convention respects the standards (lowercase, dash in names, starting with letter, etc.) Current version of i.js does not support self-closing tags like <my-first-component/>. All tags must have an opening and closing tags <my-first-component></my-first-component> You would need named components only if you would like to make some new tags for a regular site and are not planning to build an SPA using the i.js ideology. If you wish to unleash the true power of i.js and use it as an application blueprint you would better use aliases. We will discuss them later.

Component state

Let’s add a default state to our component.

Component state is a regular JavaScript object like

const state = {
    name: 'Dmitry',
    prefer: ['apples', 'lemons', 'milk', 'sugar'],
    profile: { age: 42 }
};

But in i.js you declare it like this

<script type="iris/state">
    name: 'Dmitry',
    prefer: ['apples', 'lemons', 'milk', 'sugar'],
    profile: {
        age: 42
    }
</script>

Every component has a state.☝️ If you do not declare a state then it is empty.

In our very first example state is an empty object.

const state = {}
  • to show any parts of state inside a template
  • to change any parts of a template depending on state values
  • to react to changes of attributes/properties of a component
  • to propagate changes of state to a parent component on parent’s demand

Known issues

Is the tag closing or is this a “greater” operator?

Do not use > inside attributes expression or if directive as the compiler would assume it as tag closure.

Examples

Working Not working How to fix
<div id=(state.a < 10)> <div id=(state.a > 10)> <div id=(10 < state.a)>
<div if(state.a < 5)> <div if(state.a > 5)> <div if(5 < state.a)>


Is this problem a major one? Not at all since any a > b expression can be switched to b < a. It is more a thing to remember especially if you are a true adept of the i.js Philosophy. Surely but not right now. Fixing it would take a substantial effort to be applied for change of a template parser logic. Next version most probably. PoC of a new compiler is being tested.

Whitespaces inside attributes expressions

Whitespaces cannot be used inside expressions in attributes unless the expression is not in brackets. The thing is that whitespace is the only sign for a compiler to determine the end of one attribute and start of another. This is done intentionally by design. Such logic allows to skip usage of brackets or quotes in many cases thus making most expressions shorter and more readable.

Examples

Working Not working How to fix
<input value=state.a+5> <input value=state.a + 5> <input value=(state.a + 5)>
